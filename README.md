# Идея продукта
## Проблема
В компании много гугл, word и pdf документов. Это правила компании, регламенты, инструкции, описания продуктов и ценообразование. Много много файлов, находящихся в разных местах. Освоить их все требуется очень много времени и опыта, что в конечном счёте приводит к тому, что сотрудники иногда про что-то забывают и делают не всё что нужно или делают неправильно.

## Решение
Сделать AI агента (онлайн), которому вначале скармливаем ссылки на документы и загружаем файлы, а далее сотрудник может задавать вопросы (и если необходимо кратко описывает свою ситуацию "контекст"), например: "Какие фото нужны для фотоотчёта по магистральным каналам?" или "За что отвечает Менеджер при организации нового канала связи?".

Агент предоставляет чёткий ответ(-ы) и пояснения на основании чего этот ответ: названия документов где об этом написано и ссылки на на эти документы (если они онлайн).

## Почему AI
Потому-что по другому это сотруднику самому нужно найти документацию в которой есть ответ, залезть в неё и найти ответ, либо отвлекать других сотрудников или руководителя с вопросом подсказать как делать правильно.

# Какие технологии использовал и почему

**Бэкенд**
- **Koa** — лёгкий Node.js-фреймворк на ES-модулях. Выбран за минимализм: не навязывает структуру, легко добавляются только нужные middleware (роутер, тело запроса, CORS).
- **SQLite + better-sqlite3** — встроенная база данных без отдельного сервера. Идеально для MVP: нулевая инфраструктура, синхронный API, весь стейт в одном файле.
- **@xenova/transformers** — запускает ML-модели прямо в Node.js без Python и без внешних API. Используется для создания эмбеддингов документов и поисковых запросов локально.
- **Модель Xenova/multilingual-e5-small** — многоязычная модель эмбеддингов (~120 МБ), хорошо понимает русский язык. Разделяет запросы (`query:`) и пассажи документов (`passage:`) для лучшего качества поиска.
- **pdf-parse** — парсит PDF-файлы и извлекает текст без внешних зависимостей.
- **mammoth** — конвертирует .doc/.docx в текст, корректно обрабатывает форматирование Word.
- **cheerio** — парсит HTML для извлечения текста из опубликованных Google Docs.
- **google-auth-library** — официальная библиотека Google для аутентификации через сервисный аккаунт, открывает доступ к приватным Google Docs через Docs API.
- **jsonwebtoken + bcryptjs** — JWT для сессий администратора, bcrypt для безопасного хранения PIN-кода.
- **OpenRouter** — роутер LLM: позволяет использовать разные модели (GPT, Claude) через один API-ключ, с автоматическим fallback если основная модель недоступна.

**Фронтенд**
- **React 18 + TypeScript** — декларативный UI с типобезопасностью. TypeScript помогает избежать ошибок при работе с API и состоянием.
- **Vite** — быстрый сборщик с HMR. Также проксирует `/api` на бэкенд в режиме разработки, убирая проблемы с CORS.
- **Tailwind CSS** — утилитарный CSS без написания отдельных файлов стилей. Быстро строить интерфейс «по месту».
- **react-router-dom** — клиентский роутинг между страницей чата (`/`) и панелью администратора (`/admin`).
- **Feature-Sliced Design (FSD)** — архитектурная методология для организации кода по фичам и слоям. Делает кодовую базу предсказуемой и масштабируемой.

# Как устроена система (архитектура, ключевые компоненты)

Система состоит из двух сервисов, которые запускаются независимо:

```
Браузер
  │
  ├── :5000  →  React-приложение (Vite)
  │               └── /api/*  →  прокси на бэкенд
  │
  └── :3001  →  Koa-сервер (REST API)
                  ├── SQLite (документы, PIN)
                  ├── Векторное хранилище (JSON на диске)
                  └── Uploads (PDF/Word файлы)
```

**Бэкенд (`server/`)** организован по слоям:
- `routes/` — регистрирует URL-маршруты и передаёт управление контроллерам
- `controllers/` — разбирает HTTP-запрос, вызывает сервис, формирует ответ
- `services/` — вся бизнес-логика: чат, документы, PIN, RAG-пайплайн
- `repositories/` — работа с SQLite (CRUD для документов и конфига)
- `lib/` — утилиты: инициализация БД, векторное хранилище, JWT, rate limiter

**RAG-сервисы (`server/services/rag/`)**:
- `indexing.service.js` — извлекает текст из файлов/ссылок, нарезает на чанки, строит эмбеддинги, сохраняет в векторное хранилище
- `embedding.service.js` — загружает ML-модель и превращает текст в числовые векторы
- `search.service.js` — превращает вопрос в вектор и ищет ближайшие чанки по косинусному сходству
- `lib/vectorStore.js` — хранит векторы в памяти, персистирует в JSON-файл на диске

**Фронтенд (`client/src/`)** организован по слоям FSD:
- `app/` — конфигурация приложения, роутинг, глобальные настройки API
- `pages/` — страница чата (`/`) и страница администратора (`/admin`)
- `widgets/` — крупные блоки: чат-виджет и виджет управления документами
- `features/` — самостоятельные фичи: `chat`, `documentManagement`, `pinGuard`
- `entities/` — типы и модели: `document`, `message`
- `shared/` — переиспользуемые компоненты (`Button`, `Card`, `Input`), хелперы, axios-инстанс

При старте сервера индекс перестраивается в фоне для всех документов в БД. При добавлении или удалении документа индекс обновляется инкрементально (только затронутый документ).

# AI-функциональность — что именно делает AI (не просто "отвечает")

AI-функциональность реализована через RAG-пайплайн (Retrieval-Augmented Generation). Вот что происходит на каждом шаге:

**1. Парсинг и извлечение текста**
При добавлении документа система определяет его тип и извлекает текст:
- PDF → `pdf-parse` читает бинарный файл и возвращает чистый текст
- Word (.docx) → `mammoth` извлекает текст с учётом структуры документа
- Google Docs → если настроен сервисный аккаунт, текст забирается через официальный Docs API; иначе документ парсится как HTML через публичный `/pub`-адрес

**2. Нарезка на чанки**
Извлечённый текст разбивается на фрагменты по 500 символов с перекрытием 50 символов. Перекрытие гарантирует, что информация на границе двух чанков не теряется при поиске.

**3. Создание эмбеддингов (векторизация)**
Каждый чанк преобразуется в числовой вектор (embedding) с помощью модели `multilingual-e5-small`, запущенной локально через `@xenova/transformers`. Внешний API для этого не используется. Чанки документов передаются с префиксом `passage:`, как требует модель.

**4. Сохранение в векторное хранилище**
Векторы вместе с текстом и метаданными (название документа, URL) сохраняются в памяти и персистируются в JSON-файл на диске. Хранилище самодостаточное, без отдельного сервера.

**5. Поиск по вопросу пользователя**
Когда пользователь задаёт вопрос, он тоже векторизуется (с префиксом `query:`). Затем вычисляется косинусное сходство между вектором вопроса и всеми векторами чанков. Возвращаются топ-5 наиболее релевантных фрагментов.

**6. Формирование промпта с контекстом**
Найденные фрагменты вставляются в промпт вместе с вопросом. Системная инструкция строго предписывает модели отвечать только на основе предоставленного контекста и честно говорить, если ответа в документах нет. В конце каждого абзаца модель указывает источник.

**7. Вызов LLM через OpenRouter**
Промпт отправляется в OpenRouter (модель `gpt-oss-20b:free`). Если основная модель недоступна (503), автоматически пробуются fallback-модели. История последних 6 сообщений диалога передаётся для контекста.

**8. Извлечение источников без галлюцинаций**
После получения ответа система сопоставляет упомянутые в тексте названия документов с реальным списком чанков, которые были переданы модели. Источники формируются только из тех документов, что действительно были в контексте — это исключает выдуманные ссылки.

# UX — как пользователь взаимодействует с AI-фичами

**Роль: сотрудник (страница `/`)**
- Открывает чат и сразу может задать вопрос в текстовое поле внизу экрана
- Пишет вопрос в свободной форме, при необходимости добавляет контекст своей ситуации
- После отправки видит ответ в виде отформатированного текста (поддерживается Markdown)
- Под ответом отображается список источников — названия документов, из которых взята информация, с кликабельными ссылками для Google Docs
- Может продолжать диалог: история передаётся в LLM, можно уточнять и переспрашивать
- Если в базе нет документов или модель недоступна, получает понятное сообщение с инструкцией что делать

**Роль: администратор (страница `/admin`)**
- При первом переходе на `/admin` появляется модальное окно с запросом PIN-кода
- После ввода верного PIN получает JWT-токен, который сохраняется в браузере — повторный вход не нужен
- Видит список всех загруженных документов с названием, типом и статусом индексации
- Добавляет документы двумя способами:
  - Загрузить файл — выбирает PDF или Word-документ с компьютера
  - Добавить ссылку — вставляет URL Google Docs; название подтягивается автоматически
- Удаляет документы из базы — файл и все его чанки удаляются немедленно
- Изменения в базе знаний сразу отражаются в чате для всех сотрудников

# Как развернуть проект локально (шаги)

**Требования:** Node.js 18+ и npm.

1. Клонировать репозиторий:
   ```bash
   git clone <url-репозитория>
   cd <папка-проекта>
   ```

2. Установить зависимости для сервера и клиента:
   ```bash
   npm run install:all
   ```

3. Создать файл `server/.env` и задать переменные окружения:
   ```env
   # Обязательно — ключ для доступа к LLM
   OPENROUTER_KEY=sk-or-...

   # Опционально — секрет для подписи JWT (по умолчанию используется дев-значение)
   JWT_SECRET=любая-длинная-случайная-строка

   # Опционально — PIN для входа в /admin (по умолчанию 1234 при первом запуске)
   ADMIN_PIN=ваш-пин

   # Опционально — для доступа к приватным Google Docs через Docs API
   # Если не задан, работает только с публично опубликованными документами
   GOOGLE_SERVICE_ACCOUNT_JSON={"type":"service_account","project_id":"..."}
   ```

4. Запустить сервер (порт 3001):
   ```bash
   npm run dev:server
   ```

5. В отдельном терминале запустить клиент (порт 5000):
   ```bash
   npm run dev:client
   ```

6. При первом старте сервер автоматически скачает ML-модель (~120 МБ) — это займёт несколько минут. В логах появится сообщение `[embedding] model ready`.

# Как попробовать: что открыть, как загрузить данные, что должно произойти

1. **Открыть приложение** — перейти в браузере на `http://localhost:5000`

2. **Зайти в панель администратора** — перейти на `http://localhost:5000/admin`, ввести PIN (по умолчанию `1234`)

3. **Добавить документ** — в форме «Добавить документ» выбрать один из способов:
   - Нажать «Выбрать файл» и загрузить PDF или Word-файл с локального компьютера
   - Вставить ссылку на Google Docs (документ должен быть открыт «всем, у кого есть ссылка»)
   - Нажать кнопку добавления и дождаться появления документа в списке со статусом «индексирован»

4. **Задать вопрос** — вернуться на главную страницу (`/`), написать вопрос по содержимому загруженного документа и нажать отправить

5. **Что произойдёт** — система найдёт релевантные фрагменты документа, передаст их как контекст в LLM и вернёт ответ. Под ответом отобразятся источники — названия документов, из которых взята информация. Для Google Docs будет кликабельная ссылка на оригинал.

6. **Проверить работу RAG** — задать вопрос, ответа на который в документах нет. Агент честно ответит, что информации не нашёл, вместо того чтобы придумывать.
